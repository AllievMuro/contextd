The Problem

AI Agents in 2026 Are Flying Blind

Every new session erases everything. There is no standard for how an agent should commit, retrieve, and attest to its own memory on Ethereum — and the workarounds are costly, brittle, and unauditable.

🧠

Radical Statelessness — Every Session Starts at Zero

ERC-4337 smart account agents deployed on-chain today carry no memory between invocations. Prior decisions, learned preferences, and multi-step context evaporate the moment a session closes. Agents repeat the same mistakes, lose conversational continuity, and cannot build compounding knowledge across workflows — making sophisticated multi-agent coordination effectively impossible.

Impact
💸

Raw Calldata Storage Costs Hundreds of Dollars Per Session

Teams that attempt to persist context on-chain by embedding raw vectors directly into calldata face brutal gas economics — storing a modest 32KB embedding in calldata costs upward of $80–$400 depending on network congestion. This makes any persistent-memory architecture economically unviable at production scale, forcing developers to choose between auditability and affordability.

Impact
🔒

Centralized Vector Databases Destroy Trust and Auditability

The most common workaround — bolting on Pinecone, Weaviate, or a proprietary vector DB — introduces a single point of failure with zero on-chain auditability. Any party can silently mutate or delete an agent's history. For financial, governance, or compliance-critical workflows, this is a non-starter. There is no cryptographic proof that the context an agent acts on hasn't been tampered with.

How It Works

How Contextd Works

A three-step protocol that turns EIP-4844 blobs into a permanent, verifiable agent memory primitive — integrated into your existing LangChain or LlamaIndex pipeline in minutes.

1

Commit — Write Compressed Snapshots as EIP-4844 Blobs

At the end of each session (or on a configurable trigger), your agent calls agent.commitContext(sessionId, embeddingVector). The TypeScript SDK compresses the semantic snapshot, constructs an EIP-4844 blob transaction, and submits it via an ERC-4337 UserOperation signed by the agent's smart account. Blob fees are ~$0.01 per commit — 100–10,000× cheaper than equivalent calldata. No architectural rewrite required; middleware adapters handle the entire flow transparently.

EIP-4844 Blobs ERC-4337 UserOp TypeScript SDK
2

Anchor — Merkle Root Registered On-Chain

Simultaneously with the blob commit, the ContextRegistry Solidity contract (under 200 lines) records a Merkle root of the snapshot keyed to the agent's ERC-4337 smart account address. This creates an immutable, tamper-evident on-chain fingerprint of the agent's memory state. Any future party — a co-agent, a dApp, or an auditor — can verify that a retrieved memory shard corresponds exactly to what was committed, without reading the raw blob data.

Solidity 0.8.x Merkle Patricia Tries ERC-7786
3

Retrieve & Verify — Trustless Context at Query Time

A Ponder-based TypeScript indexer tracks all ContextRegistry events and exposes a REST API for Merkle proof retrieval. Downstream consumers — another agent, a frontend dApp, or a compliance system — request a memory shard and receive it alongside a Merkle proof. Verification is performed against the on-chain root in a single contract call, confirming the context is authentic and unmodified. For blobs beyond the 18-day EIP-4844 window, an IPFS/Filecoin archival pipeline ensures permanent availability.

Features

Everything You Need for Agent Memory

Contextd is designed as a drop-in protocol primitive, not a platform lock-in. Every component is open, composable, and auditable.

⛓️

Sub-Cent On-Chain Commits

Contextd exploits EIP-4844's proto-danksharding blob fee structure — the same economics designed for rollup data — to store compressed semantic snapshots for approximately $0.01 per session commit. At 1,000 commits per day, that's $10 in total memory costs versus $80,000+ with raw calldata. The economics that made persistent agent memory impossible are gone.

🔐

Cryptographically Verifiable Context

Every memory shard is signed by the agent's ERC-4337 smart account and anchored via Merkle root in the ContextRegistry contract. Any consumer can call the on-chain verification endpoint to confirm that a retrieved snapshot is byte-for-byte identical to what the agent committed — with no trust assumptions, no third-party oracle, and no possibility of silent tampering. Auditability is a first-class property, not an afterthought.

🔌

LangChain & LlamaIndex Adapters

Existing agent pipelines adopt Contextd with a single configuration flag — no architectural rewrites. The LangChain TypeScript adapter wraps your memory chain transparently, while the LlamaIndex Python adapter provides a ContextdMemoryStore class compatible with LlamaIndex's storage interface. Developers already building with the most popular agent frameworks can be live on Contextd in under an hour.

🤝

Multi-Agent Context Sharing

Contextd's co-agent query interface allows a second ERC-4337 smart account to trustlessly read and verify another agent's committed context root on-chain. This enables complex multi-agent workflows where agents can share verified knowledge, delegate context, and build on each other's session history — without any centralized orchestration layer or trusted intermediary. True composable agent memory for DAOs, protocols, and automated pipelines.
